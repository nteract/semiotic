# Semiotic Docs

This directory contains the source for the Semiotic documentation site. It is part of the main repository and uses the same package scripts as the library.

## Local Development

Install dependencies from the repository root:

```bash
npm install
```

Start the docs site:

```bash
npm run docs:dev
```

You can verify the docs entry is healthy after config changes with:

```bash
npm run check:docs-entry
```

Build the production site:

```bash
npm run website:build
```

The Vite website build uses `docs/public/index.html` as its source and writes the built site to `docs/build`. After Vite emits the interactive SPA bundle, `scripts/prerender.mjs` creates one HTML file per route and embeds sanitized, route-specific machine-readable content in each page's `<noscript>` fallback. It also writes `docs/build/llms-routes.json`, a structured manifest with route text, headings, code blocks, and links for agents that prefer JSON over HTML.

## Current Structure

```text
docs/
  public/                 # static assets and generated public docs data
  src/
    App.jsx               # app shell and route wiring
    Landing.js            # landing page
    components/           # shared docs components
    examples/             # reusable example components
    pages/                # feature, chart, server, and playground pages
    process.js            # prop metadata grouping for generated examples
```

Useful entry points:

- `docs/src/App.jsx` for route wiring, redirects, and page registration.
- `docs/src/pages` for most authored documentation pages.
- `docs/src/components` for shared docs UI.
- `docs/src/process.js` for prop grouping used by generated examples.
- `scripts/generate-component-descriptions.mjs` and `typedoc` for generated API data.

## Import Guidance

Docs examples should teach the import path that matches the example:

- Single XY chart: `semiotic/xy`.
- Single ordinal chart: `semiotic/ordinal`.
- Single network chart: `semiotic/network`.
- Geo chart: `semiotic/geo`.
- Realtime-only chart: `semiotic/realtime`.
- Mixed dashboards or examples that intentionally combine chart families: `semiotic`.
- Validation, serialization, and helper examples: `semiotic/utils` when no chart family import is already clearer.

The root `semiotic` import is supported. Prefer subpaths in focused examples because they teach bundle-aware usage.

## Adding or Updating Pages

1. Add or edit the page under `docs/src/pages`.
2. Register the route in `docs/src/App.jsx` when adding a new page.
3. Reuse existing docs components from `docs/src/components` where possible.
4. Keep examples executable and aligned with the current HOC-first API.
5. Include data provenance for non-trivial datasets.
6. Run the relevant docs and release checks before opening a PR.

Common checks:

```bash
npm run website:build
npm run check:docs-routes
npm run check:chart-specs
npm run check:ai-examples-coverage
```

For visual changes, also run the Playwright workflow that covers the affected examples:

```bash
npm run dist
npm run test:dist
```

## Generated Content

Some documentation artifacts are generated or synchronized from source files:

- API JSON and component descriptions come from `npm run docs:api:json`.
- AI behavior contract sections are checked with `npm run check:ai-contracts` and regenerated with `npm run docs:ai-contracts`.
- Route-level machine-readable HTML and `llms-routes.json` are generated by `npm run website:build` after the Vite docs build.
- `CLAUDE.md` is the primary assistant instruction source. Regenerate its mirrored AI instruction files with `npm run docs:ai-instructions`; verify them without writing with `npm run check:ai-instructions`.

Avoid hand-editing generated output unless the corresponding source cannot reasonably be updated.
